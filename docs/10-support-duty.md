# 10 — Поддержка и дежурство менеджера

Онлайн-чат клиент ↔ дежурный менеджер, **релей через бота** (нет обмена прямыми
Telegram-контактами). Менеджер видит личность клиента (имя/телефон и его заказы);
клиент видит менеджера как **имя + телефон дежурного**, без прямого Telegram. Тред
переписки хранится в Postgres.

## Рабочее время отделения — dev-конфиг

Девы **один раз** задают часы работы **по каждому дню недели** в конфиге.
**Нигде в UI не настраивается и никому, кроме dev, не видно.** Авто-применяется
(в т.ч. для клиента). Europe/Kyiv.

## Клиент → 💬 Звернення до менеджера

1. Клиент жмёт 💬 Звернення до менеджера.
2. Бот проверяет расписание и дежурного:
   - **рабочее время + есть дежурный** → имя дежурного, его часы и **телефон**,
     кнопка «Почати чат»;
   - **вне рабочего времени** → «відділення зараз не працює», обращение в очередь
     (`waiting`), ответ на следующий рабочий день;
   - **рабочее время, но нет дежурного** → обращение в очередь **+ пинг
     менеджерам** с правом `can_handle_support` (заступить на дежурство). Владелец
     дежурным быть не может (`DUTY_ROLES = {manager}`), поэтому очередь без
     дежурного пингует менеджеров, а не владельца.
3. Клиент пишет → сообщение релеится дежурному (тред `SupportThread`).
4. Переписка через бота. Тред закрывает менеджер (кнопка «✅ Закрити») **или сам
   клиент** кнопкой «⬅️ Завершити чат» (тред → `closed`, дежурный уведомлён).
5. Кнопка «Зв'язатися з менеджером» из push (нестандартные ситуации) открывает
   этот же тред (привязка к ТТН).

## Менеджер → дежурство и чат

1. Начало смены: нажатие **«🟢 Я на зв'язку»** = открытие сессии-смены +
   включение дежурства (`on_duty`) — это и есть утренняя авторизация менеджера на
   день. `/start` только показывает меню; смену открывает лишь эта кнопка. Все
   обращения маршрутизируются ему. (Сессии — [03-roles-permissions.md](03-roles-permissions.md).)
2. Получает обращения (push) → отвечает в треде; видит профиль клиента и заказы.
3. Может **закрыть тред** (`closed`).
4. **Конец смены — автоматически:** дежурство снимается (`on_duty` off) при
   **закрытии отделения** (конец рабочих часов в едином расписании). Europe/Kyiv.
   Кнопки выключения у менеджера нет.
5. Если на дежурство никто не встал — обращения копятся, менеджеры (с
   `can_handle_support`) уведомлены заступить. **Для локального теста релея вне
   графика** проще всего временно расширить `WORK_SCHEDULE` в `.env` на текущий
   день (тогда офис «открыт», менеджер встаёт на дежурство штатно, и воркер его не
   снимает).

## Лог переписок (владелец / dev)

- Все треды и сообщения **в Postgres** (`SupportThread`, `SupportMessage`) —
  ничего не теряется.
- **Владелец и dev** видят **полный лог**: список обращений (клиент, дежурный
  менеджер, статус `open/waiting/closed`, даты) → тред показывает всю переписку.
  Поиск (клиент/менеджер/дата), пагинация.
- Менеджер видит свои/назначенные треды; полный лог по всем — только owner/dev.

## Модель данных

- `SupportThread` — client_id, assigned_manager_id, status (open/waiting/closed),
  привязка к ТТН (опц.), таймстемпы.
- `SupportMessage` — thread_id, sender_role, text, created_at.
- Дежурство — поля у менеджера (`on_duty`, `duty_date`).
- Рабочее время — **dev-конфиг** часов по дням недели (нигде в UI), Europe/Kyiv;
  телефон — из профиля дежурного.
