# 09 — Интеграция Нової Пошти

Модуль `app/novaposhta/` (client, methods, schemas, tracking, exceptions). Ключ
НП — у каждого ФОП клиента, зашифрован Fernet в Postgres. Бот — единственный
канал отправок.

## Используемые методы

- **`InternetDocument.save`** — создание ТТН (NP-first: сначала save, потом
  `Shipment` + резерв).
- **`InternetDocument.delete`** — отмена ТТН (до `dispatched`).
- **`InternetDocument.getDocumentPrice`** — расчёт стоимости/срока (онлайн-
  ценообразование).
- **`TrackingDocument.getStatusDocuments`** — трекинг (батч до 100).
- **`Address.getCities` / `Address.getWarehouses`** — справочники городов/
  отделений (кэш в Redis с TTL).
- **`validate_key_and_get_sender`** — проверка ключа ФОП + подтягивание Ref
  контрагента/контакта при сохранении ФОП.

## Поля SenderProfile (ФОП) → отправитель ТТН

- `client_id` — владелец ФОП (FK на users).
- `name` — назва ФОП (видна клиенту при выборе).
- `np_api_key` — ключ НП, **Fernet-шифр** (для вызовов API).
- `sender_full_name` — ПІБ отправителя (контрагент НП).
- `sender_phone` — телефон отправителя.
- `org_type` — ФОП / ТОВ.
- `edrpou` — ЄДРПОУ/ІПН (для ТОВ/ФОП по необходимости).
- `np_sender_ref` / `np_contact_ref` — Ref контрагента-отправителя в НП
  (кэшируются при валидации ключа).
- `np_sender_warehouse` — склад-отправитель НП (наш склад по умолчанию).
- `is_default` — ФОП по умолчанию (если один — авто-выбор).
- `created_at`, `updated_at`.

При сохранении ФОП: валидируем ключ, подтягиваем Ref'ы и реквизиты, кэшируем.
Ошибка ключа → ФОП не сохраняется.

## Поля оплаты/страховки → `InternetDocument.save` (решение F)

- `PayerType` = **Recipient** (по умолчанию) или **Sender** — **выбор клиента**
  на шаге 12 флоу ТТН. (Платник за доставку.)
- `PaymentMethod` = **Cash** — захардкожено, клиенту не показывается; на функции
  бота не влияет (меняется одной константой).
- `Cost` = **оцінкова (страхова) сума**, клиент вводит вручную (шаг 15); база
  страхового сбора; валидация > 0 и ≥ минимума НП.
- **Накладений платіж (COD)** — только если выбран на шаге 13:
  `BackwardDeliveryData = [{ PayerType: Recipient, CargoType: 'Money',
  RedeliveryString: <сума накладеного> }]` — комиссию платит **отримувач**.
- Передоплата → `BackwardDeliveryData` не отправляется.

## Получатель

- Тип: **фізособа** (PrivatePerson) / **юрособа** (Organization).
- Юр: назва ФОП/ТОВ + ЄДРПОУ при необходимости; контактна особа (ПІБ + телефон).
- Місто/відділення — из справочников НП по выбору клиента.

## Синхронизация ценообразования (онлайн)

Все данные, влияющие на стоимость доставки при формировании ТТН (тарифы, вес/
градации, габариты, відстань місто↔відділення, страховой сбор от `Cost`, комиссия
COD, доступные відділення/міста), **берутся из актуального API НП онлайн**
(`getDocumentPrice`, `Address.*`), а **не** из локальных констант. Справочники
кэшируются в Redis с TTL; ценовые значения сверяются с НП на момент создания ТТН —
бот не хранит «замороженных» тарифов.

## Расхождение веса при отправке

Декларируемый размер — оценка. При сдаче НП перевзешивает, **сама пересчитывает
стоимость и отправляет посылку** — без отмены/пересоздания и без беспокойства
клиента. Платит платник (по умолчанию получатель). Бот может тихо синхронизировать
факт. вес/стоимость в карточку ТТН для учёта (без сообщений клиенту).

## Надёжность

NP-first порядок (save → запись + резерв); ретраи к НП; транзакционность списания;
понятные ошибки клиенту при сбое (ничего не резервируем).
