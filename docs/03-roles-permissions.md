# 03 — Ролевая модель и доступ

## Иерархия ролей

Enum (строковые значения, не менять после первой миграции):
`client` < `manager` < `owner`. Роль `operator` **отложена** (добавим миграцией
при необходимости).

- **client** — личный кабинет: товары (только просмотр), статистика, создание
  ТТН (ФОП настраивает персонал), свои отправления, настройки уведомлений,
  поддержка (чат с дежурным менеджером).
- **manager** — вся операционка: приёмка товара (Google-лист), обработка/
  отправка ТТН (push по каждой новой ТТН), поддержка клиентов (дежурство, чат,
  возвраты/техпроблемы), управление клиентами (подтверждение/блокировка/
  удаление), отчёты. Без управления другими менеджерами.
- **owner** — всё, что менеджер + управление менеджерами, подтверждение
  регистраций, аналитика.

Управление строго сверху вниз: `can_manage(actor, target)` → роль actor строго
выше target. Менеджеры друг другом не управляют; назначает/снимает менеджеров
только владелец.

## Клієнтський акаунт

`User` — фактична людина й Telegram-акаунт, а `ClientAccount` — бізнес-контекст.
ФОП, ТТН, рухи складу, підтримка й статистика належать `account_id`; у ТТН
окремо зберігається `created_by_user_id`, а в повідомленні підтримки —
`sender_user_id`.

Членство (`client_account_memberships`) має ролі `account_owner` і `employee`.
Власник керує командою, ФОП, реквізитами та ключами НП. Працівник має весь
операційний кабінет акаунта, але не керує командою, ФОП або ключами. Унікальний
`user_id` membership забороняє належність до двох акаунтів.

Запрошення створює користувача без `telegram_id`; активація відбувається після
збігу номера з Telegram-контактом. `require_account_member` перевіряє одночасно
користувача, акаунт і membership, а `require_account_owner` додає гейт власника.
Заблокований працівник втрачає доступ негайно, але історичні ТТН, підтримка та
аудит не видаляються.

## Гранулярные права менеджера (per-flag)

У менеджера — набор флагов в `users.permissions` (JSONB), напр.
`can_edit_clients`, `can_manage_clients`, `can_export_reports`. **Владелец и dev
выдают/отзывают** эти флаги конкретному менеджеру. По умолчанию флаги включены;
проверка — `has_permission(user, flag)`. Каждое изменение → `audit_logs`.

## Авторизация

**Только телефон, для всех ролей.** Личность даёт сам Telegram (проверенный
аккаунт) + `request_contact` (Telegram-verified номер). Доступ гейтится ролью и
ручным подтверждением.

## Сессии

- **Клиент** — без сессий: после подтверждения залогинен постоянно, до
  блокировки.
- **Менеджер** — сессия = смена. Утром открывает бота и жмёт **🟢 Я на зв'язку**
  — это «авторизация на день» + включение дежурства; `/start` лишь показывает
  меню, смену открывает **только** эта кнопка. Авто-истечение сессии при закрытии
  отделения (по расписанию `work_schedule`, Europe/Kyiv). См.
  [06-flows-manager.md](06-flows-manager.md), [10-support-duty.md](10-support-duty.md).
- **Owner / dev** — сессия/TTL отложены (возможно позже). Для защиты
  чувствительных действий — step-up PIN/OTP на конкретное действие, без
  blanket-logout.
- **Хранение** — Redis (TTL по `telegram_id`); проверка в middleware. Логика
  проектируется переиспользуемой (в будущем UI — Telegram Mini App).

## Dev god-mode

Отдельный механизм поверх ролей. **`DEV_TELEGRAM_IDS`** в `.env` — allowlist
dev-аккаунтов. Для этих ID:

- полный просмотр и правка данных/операций на всех уровнях;
- команда **`/as client|manager|owner`** — рендер меню и флоу любой роли на
  своём аккаунте (предпросмотр), без отдельных Telegram-аккаунтов;
- **impersonation** (`/as_user <id|телефон>`) — «смотреть как конкретный
  пользователь» (его данные/экраны);
- все dev-действия пишутся в `audit_logs` (action `dev_*`).

**Реализация:** `permissions.py` проверяет dev-allowlist **первым**;
`handlers/dev.py` держит `/as`, `/as_user`, `/as off`; middleware
прокидывает «эффективную роль» и «эффективного пользователя» из dev-контекста
(хранится в FSM/Redis).

Подробные сценарии dev — [07-flows-owner-dev.md](07-flows-owner-dev.md).
