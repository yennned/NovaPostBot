# 07 — Владелец и Dev god-mode

## Меню владельца (uk)

Всё, что у менеджера (доступ ко всем тем же разделам), **плюс**:

- 👔 **Персонал** — управление менеджерами (ниже).
- 📈 **Аналітика** — динамика по периодам, по менеджерам, поддержка, финансовый
  отчёт (fee-формула), опоздавшие ТТН по SLA (ниже).

Раздела «Налаштування системи» нет: расписание/часы работы нигде в UI — только
dev-конфиг.

## 👔 Персонал (только владелец и dev)

Менеджер этот раздел не видит (`can_manage(owner, manager)`). Все действия → audit.

- **Сценарий 1. Список персонала.** Менеджеры: ПІБ, телефон, статус, 🟢 на зв'язку,
  набор прав (флаги). Поиск, пагинация.
- **Сценарий 2. Добавить менеджера.** Владелец даёт доступ **по телефону либо по
  Telegram-ID** (НЕ из активных клиентов) → роль `manager`. По умолчанию все
  per-flag права включены. Уведомление новому менеджеру + audit.
- **Сценарий 3. Карточка менеджера.** Профиль, статус дежурства, сводка по работе
  (обработано ТТН, возвраты — из аналитики), список прав. Действия: ✏️ Права ·
  Заблокувати/Розблокувати · Зняти роль.
- **Сценарий 4. Права (per-flag).** Тумблеры флагов `users.permissions`
  (`can_edit_clients`, `can_manage_clients`, `can_export_reports`…). Владелец/dev
  выдают/отзывают; применяется сразу; `has_permission(user, flag)` гейтит. Audit.
- **Сценарий 5. Блокировка / снятие роли.** «Заблокувати» → `status blocked`;
  «Розблокувати» → вернуть. **Зняти роль** → роль менеджера снимается (доступ к
  меню менеджера закрыт), дежурство/треды снимаются. Audit. Назначать/снимать
  менеджеров может только владелец.

## 📈 Аналітика (только владелец)

В дополнение к отчётам менеджера (сводка, по клиенту, склад):

- **Динамика по периодам.**
- **По менеджерам** — сколько обработал, возвраты.
- **Сводка по поддержке** — обращения откр./закр.
- **Финансовый отчёт.** Комиссия за сборку заказа:
  `fee(ТТН) = 20 грн + (кількість одиниць − 1) × 1 грн` (первая единица — 20 грн,
  каждая следующая +1 грн; покрывает сборку/упаковку/ТТН/передачу). За период
  (день/неделя/месяц): количество ТТН, количество единиц, сумма fee — минус
  бесплатные по SLA (промах >30 хв → fee = 0, см.
  [08-notifications-tracking-returns.md](08-notifications-tracking-returns.md)).
  (Суммы наложенного/стоимость доставки **не считаем**.)
- **Опоздавшие ТТН (SLA >30 хв):** количество + список номеров ТТН (они же
  бесплатные).

`reports.py` даёт frozen-dataclass агрегаты; рендер — в хендлере.

## 🧪 Dev god-mode (флоу)

Только для Telegram-ID из **`DEV_TELEGRAM_IDS`** (allowlist).
`permissions.py` проверяет dev-allowlist **первым**. Все dev-действия → `audit_logs`
(`dev_*`). Эффективная роль/пользователь — в FSM/Redis, прокидываются middleware.

- **Сценарий 1. Полный доступ.** Просмотр и правка данных/операций на всех уровнях
  (клиенты, ФОП/ключи, ТТН, склад-коррекции, поддержка, персонал), включая
  защищённые действия владельца.
- **Сценарий 2. `/as <role>`.** `/as client|manager|owner` → рендер меню и флоу
  роли на аккаунте dev; индикатор «🧪 as manager»; `/as off` — назад в god-mode.
- **Сценарий 3. Impersonation.** `/as_user <id|телефон>` → подстановка
  эффективного пользователя (его меню/данные/экраны); удобно воспроизводить баги;
  `dev_impersonate` в audit; `/as off` — выход.
- **Сценарий 4. Аудит.** Любое изменение под god-mode/impersonation — в `audit_logs`
  с реальным dev-ID, типом `dev_*`, до/после. (Пуши владельцу/dev в нестандартных
  ситуациях не шлются.)
