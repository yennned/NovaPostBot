# 08 — Уведомления, трекинг, возвраты и нестандартные ситуации

## Матрица уведомлений по ролям

Всё настраиваемо вкл/выкл, кроме гарантированных.

- **Клиент:** регистрация подтверждена; статусы ТТН/трекинг
  (создана/зарегистрирована/отправлена/доставлена/возврат); изменение остатков и
  низкий остаток.
- **Менеджер:** **каждая новая ТТН** (push для обработки/отправки); запросы
  клиентов (поддержка, возвраты/техпроблемы) дежурному; отмена ТТН клиентом;
  новый клиент на подтверждение (кнопки); эскалации/ошибки НП; низкий остаток.
- **Владелец:** новый клиент на подтверждение (гарантированно, из
  `OWNER_TELEGRAM_IDS` + БД); сводка/аналитика; критичные события (блокировки,
  сбои интеграции); обращение без дежурного; + всё, что у менеджера.

Единый сервис `notifications.py` с «тихой» обработкой ошибок Telegram и дедупом
получателей.

## Трекинг (воркер)

Периодический опрос НП (`TrackingDocument.getStatusDocuments`, батч до 100) по
ключу клиента; продвижение статуса, пуш клиенту, списание резерва при переходе
через `dispatched`.

Статусы ТТН: `created → confirmed → registered → dispatched → in_transit →
arrived → delivered`; ветки: `returning`/`returned`, `lost`/`damaged`,
`cancelled`.

### SLA («Відправка за 30 хв»)

Правило: ТТН должна уйти (`dispatched` по трекингу НП) в течение **30 рабочих
минут** от создания клиентом. Счёт идёт **только в рабочее время отделения**
(расписание — dev-конфиг `work_schedule` по дням недели, Europe/Kyiv,
[02-architecture.md](02-architecture.md)); вне рабочих часов таймер не идёт.

- **Расчёт дедлайна.** Helper `add_working_minutes(start, 30, schedule)` →
  `sla_deadline`. Если ТТН создана вне рабочих часов — таймер стартует с начала
  ближайшего рабочего дня. *Пример:* отделение 08:00–20:00, ТТН оформлена в 21:00
  → счётчик стартует в 08:00 след. дня → `sla_deadline` = 08:30 след. дня. Если до
  закрытия осталось <30 мин — остаток догорает со следующего открытия.
- **Фиксация.** Воркер при переходе ТТН в `dispatched` сравнивает
  `dispatched_at` с `sla_deadline`: успел → `sla_met`, опоздал → `sla_missed`.
- **Промах.** При `sla_missed` комиссия за заказ обнуляется: `fee_amount = 0`,
  `fee_free = true` (обработка бесплатно). Идёт в финотчёт владельца и в список
  опоздавших ТТН ([07-flows-owner-dev.md](07-flows-owner-dev.md)).
- **Опционально:** push менеджеру при приближении дедлайна.

## Что такое «нестандартная ситуация»

**Happy-path** (вмешательство не нужно):
- ТТН: `created → registered → dispatched → in_transit → arrived → delivered`.
- Приёмка: товар пришёл в полном объёме, без брака.
- Остаток меняется штатно: приёмка (+), резерв/отгрузка (−).

**Нестандартная ситуация** — любое отклонение, требующее внимания персонала.
Бот распознаёт (трекинг/приёмка/вручную), помечает статусом/флагом, шлёт push
**клиенту и менеджеру** (с кнопкой «Зв'язатися з менеджером») и открывает право
правок **менеджеру / владельцу / dev**. Категории:
1. **Возврат** — отказ получателя; просрочка хранения → `returning`/`returned`.
2. **Утрата/повреждение** — потеряна/повреждена/не доехала → `lost`/`damaged`.
3. **Проблемы приёмки** — брак входящего, недостача, повреждение при поступлении.
4. **Сбой при создании ТТН** — ошибка НП API, неверные данные получателя.
5. **Учётные расхождения** — нехватка остатка под заказ; факт ≠ система.
6. **Зависание** — ТТН долго не двигается.

**НЕ нестандартная** (бот разбирается сам, клиента не беспокоим):
- расхождение фактического веса при отправке — НП пересчитывает и отправляет;
- штатные переходы статусов трекинга.

**Пуши и связь:** push идёт **только клиенту и менеджеру** (владельцу и dev — НЕ
шлём, хотя они могут вносить правки). В push клиенту — кнопка «Зв'язатися з
менеджером» (тред поддержки с дежурным).

## Возвраты и проблемные отправления (менеджер)

Доп. статусы: `returning`, `returned`, `lost`, `damaged`. Меняются авто по
трекингу или вручную менеджером/владельцем/dev — всегда с записью в `audit_logs`.

### A. Возврат товара

Триггеры: (1) отказ получателя, (2) просрочка хранения. НП возвращает на наш склад.
1. Бот по трекингу ловит статус возврата → `returning`. **Менеджер получает
   уведомление об отказе/возврате** («посилка повертається — забрати і повернути
   товар на склад»).
2. Посылка физически приезжает на склад.
3. Менеджер на карточке ТТН жмёт **«Повернення замовлення»** → **приём как
   повторная приёмка через бота** (состав известен из исходной ТТН):
   - бот показывает позиции исходной ТТН для проверки;
   - менеджер по каждой отмечает **ціле/придатне** или **брак/пошкоджено**;
   - подтверждение: годные → `+qty` в лист «Склад» (`ttn_return` в Postgres +
     «Історія»); брак → в склад не идёт, фиксируется, уведомление клиенту.
4. Статус → `returned`; audit. Клиент видит обновлённый остаток.

### B. Утрата/повреждение в пути

1. Бот ловит проблемный статус **или** менеджер помечает вручную.
2. Статус → `lost` / `damaged`. **Товар в остаток НЕ возвращается.**
3. Push клиенту и менеджеру.
4. Клиент оформляет **претензию в НП** (компенсация по оцінковій/страховій сумі).
5. Менеджер/dev добавляет примечание/номер претензии (audit).

## Принцип правок в базе

Любая корректировка количества — **только через журнал движений**
(`stock_movements`, тип `manual` + комментарий + автор), без прямого
редактирования остатка. Смена статуса ТТН — авто по трекингу или ручной override
менеджера/владельца/dev с записью в `audit_logs`. Бот и база консистентны.
